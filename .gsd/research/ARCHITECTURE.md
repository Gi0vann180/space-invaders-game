# Architecture Research

**Domain:** Android-first release architecture for an existing React + Canvas arcade game
**Researched:** 2026-03-09
**Confidence:** HIGH

## Recommendation

Keep the game as a web-first runtime with a thin native Android shell. Do not move gameplay into React, and do not move core gameplay into native Android code. The next-stage architecture should formalize four boundaries that are already emerging in the codebase: `app shell`, `game runtime`, `player/profile domain`, and `platform shell`.

The current project already has the correct core split for a solo developer: React owns menus, overlays, settings, and commerce-like screens; the imperative engine owns the simulation loop and Canvas rendering. The right move is to harden that split, not replace it with an ECS rewrite, React-driven gameplay, or a heavy backend.

For Android-first polish and later expansion, add a dedicated mobile platform layer around the existing web app. That layer should own app lifecycle, safe-area and system UI behavior, haptics, splash/loading readiness, back-button policy, packaging, and release automation. The engine should remain portable and unaware of Capacitor or Android specifics.

## Standard Architecture

├───────────────────────────────────────────────────────────────────────┤
│ Engine commands │ selectors │ view models │ session store            │

┌─────────────▼───────────────────────────────▼─────────────────────────┐

├───────────────────────────────────────────────────────────────────────┤

### Recommended Component Boundaries

| Component | Responsibility | Recommendation |
|-----------|----------------|----------------|
| `app-shell` | Navigation, screen composition, HUD, overlays, safe-area layout | Keep in React. No simulation logic here. |
| `game-session` | Single public interface between UI and runtime | Add a facade with commands like `startRun`, `pauseRun`, `continueStage`, `applyUpgrade`, `openShop`, `syncProfile`. |
| `game-runtime` | Fixed loop, world state, rendering, entity/system orchestration | Keep imperative and isolated from React. Split into smaller submodules, but keep one runtime owner. |
| `progression-domain` | Meta progression, unlocks, currencies, upgrade trees, run rewards | Move out of ad hoc UI/store mutation into explicit domain services and reducers. |
| `profile-domain` | Settings, consent, accessibility, player identity, local profile | Separate from volatile run state. |
| `persistence` | Repositories for profile, run history, settings, scores, sync queue | Keep IndexedDB behind repository interfaces. |
| `platform-mobile` | Capacitor app lifecycle, splash, haptics, status bar, back button, orientation, insets | Thin adapter layer. Never let runtime import Capacitor directly. |
| `network-integrations` | Optional leaderboard and future low-cost async sync endpoints | Keep best-effort and non-blocking. |
| `build-packaging` | Web build, Capacitor sync, Android signing, assets, manifests | Separate deployment concern from app logic. |

### Structure Rationale

- `app/` should own startup and shell behavior, because mobile readiness depends on when the app is truly interactive, not just when React mounted.
- `features/` should hold user-facing application flows, not raw gameplay internals.
- `game/` should remain the authoritative gameplay runtime and render stack.
- `platform/` is the critical new boundary for Android-first release work.
- `data/` should isolate persistence and remote calls so future sync or cloud save can be added without touching gameplay systems.
- `android/` should remain a thin shell project. Treat it as packaging and native integration, not the place where game rules live.

## Architectural Patterns

### Pattern 1: Runtime Facade Over Engine Internals

**What:** React talks to a stable command/query facade, not directly to engine internals or loose stores.

**When to use:** Immediately. This is the main stabilization step before adding mobile shell features.

**Why:** Today the current store bridge works, but expansion will get messy if screens, shop flow, and lifecycle handling keep mutating shared state from multiple directions.

**Trade-offs:** Slightly more boilerplate, much less coupling.

**Example:**

```typescript
export type GameSessionFacade = {
  getSnapshot(): SessionViewState
  subscribe(listener: (state: SessionViewState) => void): () => void
  startRun(): void
  pauseRun(reason: 'ui' | 'visibility' | 'app-background') : void
  resumeRun(source: 'ui' | 'app-foreground'): void
  applyShopChoice(input: ShopChoice): void
}
```

### Pattern 2: Separate Run State From Profile State

**What:** Keep volatile session state and persistent player profile state in separate stores and repositories.

**When to use:** Before deeper progression, achievements, daily rewards, cosmetics, or cloud sync.

**Why:** A mobile game evolves faster in profile/meta systems than in the frame loop. Tying both to one generic store causes accidental resets, migration pain, and UI bugs.

**Trade-offs:** More explicit data mapping, but cleaner saves and migrations.

**Recommended split:**

- `session state`: score, lives, stage, current drops, boss state, active run modifiers, transient HUD data.
- `profile state`: unlocked stages, currencies, permanent upgrades, settings, consent, cosmetic unlocks, last-sync metadata.

### Pattern 3: Platform Adapters, Not Platform Conditionals Everywhere

**What:** Wrap Capacitor plugins and browser APIs behind one platform interface.

**When to use:** As soon as the Android shell is introduced.

**Why:** Mobile polish features multiply quickly: haptics, splash readiness, status bar, pause/resume, hardware back, deep links, install/update behavior. Sprinkling `Capacitor.isNativePlatform()` across the app will rot the codebase.

**Trade-offs:** Small abstraction layer to maintain, major reduction in branching noise.

**Example:**

```typescript
export type MobilePlatform = {
  onPause(listener: () => void): () => void
  onResume(listener: () => void): () => void
  onBackPress(listener: () => boolean): () => void
  signalReady(): Promise<void>
  impact(kind: 'light' | 'medium' | 'heavy'): Promise<void>
  getInsets(): Promise<SafeAreaInsets>
}
```

### Pattern 4: Repository + Migration Boundary For Local Data

**What:** Keep IndexedDB access in versioned repositories with explicit schema migration ownership.

**When to use:** Before adding more progression, inventory, run history, or offline queueing.

**Why:** The current IndexedDB helper is fine as a primitive, but the next stage needs profile migrations, partial writes, and future-proof save formats.

**Trade-offs:** More code than direct object store access, much safer for long-lived mobile installs.

## Explicit Data Flow

### 1. User-facing gameplay loop

```text
Touch / keyboard / gamepad / Android back intent
    -> input adapters
    -> runtime input snapshot
    -> fixed update loop
    -> gameplay systems mutate world state
    -> runtime emits session snapshot
    -> session facade publishes view state
    -> React HUD / overlays / shop / results render
    -> user action dispatches command back to facade
```

### 2. Progression and meta loop

```text
Stage clear / run over / reward event
    -> progression service computes rewards, unlocks, offers
    -> profile repository persists durable changes
    -> profile store updates
    -> UI reflects unlocks, currencies, upgrade levels
    -> optional sync queue posts leaderboard or cloud-safe events later
```

### 3. Mobile shell lifecycle flow

```text
Capacitor App pause/resume/appStateChange/backButton
    -> platform-mobile adapter
    -> app lifecycle coordinator
    -> session facade pause/resume/back policy
    -> runtime loop pause/resume
    -> shell updates overlays, audio state, and navigation behavior
```

### 4. Persistence flow

```text
Bootstrap
    -> app bootstrap loads settings/profile/device preferences
    -> app shell renders minimal loading state
    -> runtime starts only after profile hydration is complete

Checkpoint
    -> profile mutations saved through repositories
    -> run summary or important milestones appended to local history
    -> sync queue retries opportunistically when network is available
```

### 5. Packaging flow for store app

```text
frontend source
    -> vite build
    -> web bundle in dist/
    -> npx cap sync android
    -> Android shell receives web assets + plugin config
    -> Android Studio / CI builds signed AAB
```

This build boundary is important: the Android project should package the already-built web game. It should not become a second application codebase.

## Packaging Boundary Recommendation

### Recommended packaging model: Capacitor shell around the web game

Use Capacitor as the app container for Android. The official Capacitor workflow is built around `build -> sync -> run/build`, and its Android runtime is explicitly meant to host web assets plus native plugins. This matches the current codebase well and avoids a rewrite into a fully native UI stack.

### What belongs in the web layer

- React shell and all menus/HUD flows
- Canvas renderer and game runtime
- Progression logic and local profile logic
- IndexedDB-backed persistence
- Optional leaderboard client and retry queue

### What belongs in the native Android shell

- App icons, splash resources, manifests, package id, signing config
- Status bar and system UI integration
- Hardware back button behavior
- App foreground/background lifecycle bridging
- Haptics, vibration, orientation lock, and safe-area handling
- Store metadata and release packaging

### What should stay out of the native shell for now

- Gameplay rules
- Shop/progression logic
- Duplicate persistence logic in Kotlin
- Native-only UI copies of menus or HUD

That boundary keeps the product expandable to iOS later, and it keeps the solo-developer cost profile under control.

## Android-first UX and Performance Architecture Notes

### Mobile UX polish modules to add

| Module | Why it matters |
|--------|----------------|
| `safe-area layout service` | Prevent clipped HUD and buttons on varied Android devices |
| `mobile input tuning` | Touch zones, drag sensitivity, tap forgiveness, thumb ergonomics |
| `haptics service` | Better hit, reward, and menu feedback |
| `readiness gate` | Controls splash hide only after profile + assets + first frame are ready |
| `back-button coordinator` | Android navigation must feel native and deterministic |

### Performance modules to add

| Module | Responsibility |
|--------|----------------|
| `device profile` | Detect low/mid/high capability buckets |
| `quality policy` | Choose particles, shader-like effects, drop caps, animation density |
| `asset loader` | Preload and stage assets so first-run is predictable |
| `frame telemetry` | Capture slow-frame stats and quality downgrades |
| `render layers` | Separate gameplay, effects, and HUD drawing concerns |

For this project size, prefer adaptive quality and disciplined frame budgeting over advanced architectural jumps like workerized rendering or a custom engine rewrite.

## Anti-Patterns To Avoid

### Anti-Pattern 1: Putting more gameplay state into React component state

**Why it fails:** It makes frame-critical logic hostage to UI re-render behavior and complicates pause/resume on mobile.

**Do this instead:** Keep runtime state imperative, publish snapshots to the UI bridge.

### Anti-Pattern 2: Letting screens mutate the shared store directly

**Why it fails:** Shop, settings, results, and progression will begin to bypass rules and create impossible states.

**Do this instead:** Route mutations through facade commands and domain services.

### Anti-Pattern 3: Mixing Capacitor calls into random UI and engine files

**Why it fails:** Packaging concerns leak everywhere and block web-only testing.

**Do this instead:** Keep all native bridging in `platform/capacitor` adapters.

### Anti-Pattern 4: Treating IndexedDB records as the domain model

**Why it fails:** Save schema ossifies too early and migrations become painful.

**Do this instead:** Define profile/run models first, then map them to repository records.

### Anti-Pattern 5: Rewriting for native or ECS before the mobile release proves the need

**Why it fails:** Huge cost, high risk, little guaranteed user value for this stage.

**Do this instead:** Harden seams, improve rendering and input, add mobile shell, ship, then profile real bottlenecks.

## Build Order Implications For The Roadmap

The roadmap should be dependency-driven, not feature-driven.

1. **Stabilize boundaries first**
   - Introduce `game-session` facade.
   - Split `session`, `profile`, and `app` state.
   - Move progression/meta writes behind repositories.

2. **Add mobile platform shell second**
   - Introduce Capacitor and Android project.
   - Add lifecycle adapter, back-button policy, splash readiness, haptics, status bar handling.
   - Keep the native shell thin.

3. **Add mobile UX and performance third**
   - Tune touch input.
   - Add safe-area aware HUD/layout.
   - Add device profile + quality policy + asset loading.

4. **Expand progression and content fourth**
   - Permanent upgrades, cosmetics, deeper rewards, better run summaries.
   - These become safer after profile and persistence boundaries exist.

5. **Add optional integrations last**
   - Leaderboards, cloud save experiments, analytics refinements, notifications.
   - Keep them optional and failure-tolerant.

This order prevents the common mistake of polishing content on top of unstable session/profile/mobile boundaries.

## Suggested Internal Boundaries

| Boundary | Communication | Rule |
|----------|---------------|------|
| `app-shell -> game-session` | commands + subscriptions | UI never imports runtime internals |
| `game-session -> game-runtime` | direct facade calls | runtime stays authoritative for frame loop |
| `progression-domain -> persistence` | repositories | domain never knows IndexedDB details |
| `platform-mobile -> app-shell` | listeners/events | mobile lifecycle enters through one adapter |
| `network-integrations -> profile/progression` | async services | network failures never block core gameplay |
