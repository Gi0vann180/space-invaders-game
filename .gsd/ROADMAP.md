# Roadmap: Space Invaders Mobile

## Overview

Roadmap focused on turning the current playable browser foundation into an Android-first mobile product by improving control comfort, combat readability, progression clarity, and release readiness in dependency order.

## Phases

**Phase Numbering:**
- Integer phases (1, 2, 3): Planned milestone work
- Decimal phases (2.1, 2.2): Urgent insertions (marked with INSERTED)

- [ ] **Phase 1: Touch Controls & Session UX** - Deliver a low-friction and comfortable touch-first gameplay loop on Android phones.
- [ ] **Phase 2: Combat Readability & Run Feedback** - Make moment-to-moment action and run outcomes clear on small mobile screens.
- [ ] **Phase 3: Progression Framing & Novelty Cadence** - Clarify progression goals and reduce repetitive run perception.
- [ ] **Phase 4: Android Packaging & Runtime Reliability** - Ship a publishable Android build with resilient lifecycle and stable performance.

## Phase Details

### Phase 1: Touch Controls & Session UX
**Goal**: Users can control and restart runs comfortably and quickly on Android touch devices.
**Depends on**: Nothing (first phase)
**Requirements**: CTRL-01, CTRL-02, CTRL-03
**Success Criteria** (what must be TRUE):
  1. User can move and shoot with touch controls that remain responsive and comfortable through complete runs on Android phones.
  2. User can configure sensitivity, touch-target behavior, and reduced-motion comfort options and feel the change immediately in gameplay.
  3. User can restart after defeat or interruption and return to active gameplay with low friction.
**Plans**: TBD

### Phase 2: Combat Readability & Run Feedback
**Goal**: Users can understand combat states and run outcomes at a glance during and after each run.
**Depends on**: Phase 1
**Requirements**: COMB-01, COMB-02, COMB-03
**Success Criteria** (what must be TRUE):
  1. User can clearly distinguish projectiles, hazards, pickups, and impact feedback during active combat on small screens.
  2. User can learn movement, risk/reward, and loop behavior by playing early runs without relying on intrusive tutorial overlays.
  3. User receives clear end-of-run feedback that explains outcome, rewards, and progress status.
**Plans**: TBD

### Phase 3: Progression Framing & Novelty Cadence
**Goal**: Users understand why progression matters and encounter enough variation to keep sessions engaging.
**Depends on**: Phase 2
**Requirements**: PROG-01, PROG-02
**Success Criteria** (what must be TRUE):
  1. User can identify what was unlocked, why it matters, and what target to pursue next.
  2. User encounters noticeable early and mid-game novelty across multiple runs, reducing repetitive-run perception.
  3. User can connect run outcomes to meaningful forward progression choices.
**Plans**: TBD

### Phase 4: Android Packaging & Runtime Reliability
**Goal**: Users can play stable sessions on Android and the project can be released through a publishable package pipeline.
**Depends on**: Phase 3
**Requirements**: ANDR-01, ANDR-02, ANDR-03
**Success Criteria** (what must be TRUE):
  1. User session survives pause/resume and common Android interruptions with reliable continuity.
  2. Team can generate a publishable Android package from the existing codebase without duplicating gameplay logic.
  3. User experiences stable performance and reliability during real play sessions on target Android devices.
**Plans**: TBD
