## Component Contract (Phase 02)

Implement or update only these UI components/states for boss encounters:

1. `BossStatusPanel`
- Location: top HUD strip or equivalent existing HUD container.
- Fields: boss name/id, HP current/max, attempt number, phase state (`telegraph`, `active`, `stagger`, `final`).
- Behavior: appears only when `bossEncounter.active === true`; updates without fade lag.

2. `BossTelegraphBanner`
- Location: upper-middle safe zone, not covering ship control visibility.
- Trigger: 1000-1500ms before high-threat boss pattern.
- Behavior: one concise warning line; auto-dismiss after action starts.

3. `BossOutcomeOverlay`
- Victory state: success color emphasis + progression-forward CTA.
- Defeat state: destructive emphasis + immediate retry CTA.
- Must preserve campaign consistency: victory continues progression, defeat offers same-stage retry.

4. `EncounterSummaryRow` (lightweight)
- Context: post-outcome panel.
- Fields: elapsed time, damage taken, remaining lives.
- Purpose: reinforce readability and perceived fairness without exposing raw internals.

## Interaction Contract

Primary focal point declaration:
- BossStatusPanel (boss HP + phase state) is the primary visual focal point during active encounter.
- BossOutcomeOverlay primary CTA is the focal point after encounter resolution.

The phase must satisfy these interaction rules, aligned with requirements CAMP-02 and CORE-04:

1. Encounter readability and fairness
- Every campaign stage (1-5) presents one boss encounter with recognizable identity.
- Telegraph precedes high-threat attacks by at least 1.0s.
- Visual warning and boss HP state must remain visible during active patterns.

2. Feedback for core outcomes
- Boss start: clear audiovisual cue sequence and state shift to boss mode.
- Player defeat during boss: immediate defeat feedback plus retry action.
- Boss victory: clear win feedback before transition to progression/shop flow.

3. State consistency
- UI reflects `bossEncounter` as single source for attempt/outcome/timing details.
- No React-only derived truth for encounter lifecycle.
- Transition screens must not conflict with existing campaign overlays.

4. Offline and telemetry-safe behavior
- Encounter UI cannot depend on network availability.
- Telemetry consent state never blocks boss UI feedback.
- Failure to log telemetry must not alter user-facing encounter flow.

5. Difficulty communication
- Difficulty should feel "quase bullet-hell, mas justo": intense but readable.
- Consecutive stage bosses must feel mechanically distinct with medium variation between attempts.
- Defeat messaging must indicate recoverability (retry now) and avoid punitive tone.
